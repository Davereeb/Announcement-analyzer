1 数据导入与断点续跑机制

每条公告在数据库中有一个状态字段，记录处理进度。重新运行时自动跳过已完成记录。

📌 数据库状态机
pending ──→ fetched ──→ done ✓
任一步骤失败 ──→ failed（retry_count < 3 可重试） 不支持格式 ──→ skipped
💡 关键设计：公告链接设为 UNIQUE 约束，导入用 INSERT OR IGNORE，多次运行不会产生重复记录。

2 文档下载与多格式文本提取

从东方财富CDN下载公告文件，自动识别文件格式，调用对应解析器提取纯文本。

⚠️ 反爬虫处理：东方财富CDN会先返回一段 JavaScript 挑战代码（而非真实PDF），需要先用 Node.js 执行该脚本提取 Cookie，再携带 Cookie 重试请求，才能拿到真实文件。

3 文本长度分级处理

根据文本字数分为三档，短文档全文送入LLM，长文档进行智能三段采样，控制Token消耗。

🟦 short（< 3,000字）

全文直接送入LLM，不作任何裁剪。
典型：简短董事会决议、股权变动公告。

🟩 medium（3,000 – 10,000字）

全文直接送入LLM，不作任何裁剪。
典型：季报、半年报摘要、重大合同公告。

🟧 long（> 10,000字）

智能三段采样，拼接送入LLM：
头部 2,000字 + [...中间省略...] + 中段 2,000字 + [...中间省略...] + 尾部 2,000字
典型：重组报告书、招股说明书、大型年报。

4 LLM 调用：System Prompt（系统提示词）

设定模型的角色和输出格式约束。保持极简，避免干扰模型对正文的判断。

 SYSTEM PROMPT 
你是A股金融公告分析专家，专注于半导体板块。
请分析公告文本，严格按JSON格式输出，不要输出任何其他内容，不要加```json标记。
💡 设计思路：System Prompt 故意写得极简——只定义角色（A股半导体分析专家）和格式约束（纯JSON，不加代码块标记）。具体的分析任务、字段定义、判断标准全部放在 User Prompt 里，这样每次调用可以携带不同的公告内容，而系统角色保持不变。

5 LLM 调用：User Prompt（用户提示词）

每条公告动态生成，包含标题、类型、正文，以及详细的分析指令和输出格式定义。

 USER PROMPT（模板） 
公告标题：{title}          # 例：江丰电子:关于2024年半年度报告的公告
公告类型：{ann_type}       # 例：业绩预告、股东减持、定增预案…

公告正文：
{text}                          # 提取后的纯文本（按文本分级处理）

---
请完成以下分析，输出纯JSON：

{
  "is_valuable": "有或无",         # 是否包含影响股价的实质信息

                                    # 重大合同、产品获批等

"summary"

string

📌 核心判断标准：价值判断 & 情感评分

以下两套标准直接写入 Prompt，是 LLM 打分的依据，也是理解结果的关键。

① 价值判断标准
输出：有 或 无
✅ 判为"有"价值
包含以下任意一类实质信息：
📈 业绩变动 📉 增减持 🔄 股份回购 🏆 重大中标 🤝 重组/并购 🏢 股权变更 💰 分红/转增 ⚖️ 重大诉讼 🚫 监管处罚 📝 重大合同 ✅ 产品获批
💡 判断核心：该信息是否会导致市场参与者重新评估公司价值或股价预期
❌ 判为"无"价值
仅包含常规程序性内容，例如：
·独立董事任职资格确认、董事/监事换届选举
·会计师事务所续聘、年报审计委托
·股东大会召开通知、会议纪要（无实质决议）
·日常资金管理授权、常规关联交易续签
·证监会/交易所问询函的格式性回复（无新信息）
·公司章程修订（技术性条款）
💡 这类公告属于合规要求的例行披露，不含增量信息
② 情感评分标准
仅对"有价值"公告打分，输出整数 -2 / -1 / 0 / +1 / +2
+2
极度正面
打分条件（满足其一）
· 重大重组/并购完成，战略意义显著
· 净利润/营收同比大幅增长（>100%）
· 重大产品获批或核心资质取得
· 大额股份回购完成，回购比例 >3%
典型案例
长川科技Q3净利润+207.6%
金海通净利润+832.58%
至正股份完成半导体资产重组
+1
正面
打分条件
· 业绩同比增长，但幅度适中（10%-100%）
· 重大中标/签署合同，金额明确
· 定增/再融资方案获批，募资顺利
· 回购进展公告，体现管理层信心
典型案例
季报业绩增长、重大订单中标
股份回购完成公告
定增方案获证监会批复
0
中性
打分条件
· 信息有实质内容，但方向不明确
· 资产出售/转让，但对整体影响有限
· 对外投资/设立子公司，规模较小
· 调研活动纪要，无明确业绩指引
典型案例
战略合作协议（金额未披露）
机构投资者调研接待公告
股权激励计划草案公示
-1
负面
打分条件
· 业绩下滑，同比下降但未亏损
· 募投项目进度延期，原因明确
· 重要股东减持计划披露
· 诉讼/仲裁立案，金额较小
典型案例
季报净利润同比下降30%
高管/大股东减持计划公告
募集资金项目延期使用申请
-2
极度负面
打分条件（满足其一）
· 营收&净利润双降，且由盈转亏
· 净利润同比暴跌（>-80%）或巨额亏损
· 重大违规/处罚，涉及核心业务
· 重大诉讼败诉，赔偿金额巨大
典型案例
欧莱新材前三季度亏损-306.5%
联动科技扣非净利润-97.39%
阿石创净利润-126.57%由盈转亏
⚠️ 重要说明：情感评分基于公告内容本身，而非当日股价走势。同一事件对不同时间窗口的股价影响可能不同，该评分反映的是"信息质量与方向"，不构成投资建议。

细粒度评分（信息密度）

📐 定义与公式

衡量一篇公告中有效信息的密集程度——同样被判为"有价值"，一份仅含2条关键数据的50页年报与一份全文都是核心数据的季报摘要，信息密度完全不同。

FORMULA
granularity = 有效信息字数 ÷ 全文字数 × 100
「有效信息字数」= LLM 估算的"对股价判断有直接参考价值"的内容字数
输出为 0–100 整数，由 LLM 基于所见文本直接给出，无需二次计算
💡 设计取舍：真正的"有效字数"需要对全文每个句子打标——计算成本极高。本方案让 LLM 在阅读文本的同时直接给出百分比估算，在精度与可行性之间取得平衡。对于 long 文档（仅送入节选片段），LLM 基于片段推断全文密度。

2 高价值 vs 低价值内容的判定

Prompt 中明确告知 LLM 哪类内容算"有效信息"，确保估算口径一致。

✅ 高价值内容（计入有效字数）
📊 关键财务数据：营收、净利润、同比增幅、毛利率
💰 金额信息：募资金额、合同金额、回购金额
📈 比率/百分比：增长率、占比、完成进度
🎯 事件结论：获批、中标、完成、违规、亏损等核心结论句
📅 关键时间节点：完成日期、生效日期
❌ 低价值内容（不计入有效字数）
📋 背景介绍与历史沿革
⚖️ 法律条文引用与合规声明
📝 模板化套语（"根据XX规定"、"经审议通过"）
🗂️ 目录、封面、签字页
📎 附录与注释（非核心数据）

4 长文本（long）的特殊说明

long 类型文档（>10,000字）只送入节选片段，granularity 基于片段推断全文，存在一定估算误差。

📐 long文本送入LLM的内容
[节选·前段正文]
  跳过封面/目录（前8%）
  取正文起始 2,000字

[...省略...]

[节选·中段正文]
  文档50%位置附近 2,000字

[...省略...]

[节选·后段正文]
  文档80%位置附近 2,000字
  跳过附录/签字（后7%）
⚠️ 已知局限
· LLM 只看到 ~6,000字，约占年报全文的 5-8%
· granularity 为基于片段的推断值，而非精确统计
· 若关键数据集中在未被采样的区段，可能低估密度

采样优化（已实施）：
· 跳过前8%封面/目录，从正文起始采样
· 跳过后7%附录/签字，从正文核心区采样
· 实测：long文本有价值率77.9%，摘要含具体数字比例82.5%

系统模块架构
⚙️

config.py 配置层

统一管理所有配置：API Key、并发数（20）、超时时长、文本分类阈值。修改一处即可调整全局行为。

🗃️

db.py 数据层

SQLite操作封装。公告链接设为唯一约束，INSERT OR IGNORE 实现幂等导入。侧表存储原始文本，主表保持精简。

⬇️

fetcher.py 下载层

核心模块。流式下载超大PDF（120s超时）；Node.js解析JS反爬虫挑战；自动识别5种文件格式并分发到对应解析器。

🤖

llm.py AI层

支持 DeepSeek / Qwen 切换。指数退避重试网络错误；正则兜底解析非标准JSON输出；长文档三段采样（头+中+尾）。

🔄

pipeline.py 调度层

asyncio.Semaphore 控制20并发；tqdm 实时进度条；每100条打印 Token 消耗与费用统计；单条异常不影响全局。

📊

exporter.py 输出层

openpyxl 生成格式化Excel：情感列条件颜色、摘要列自动换行、冻结首行、自动提取股票名称列。

遇到的技术难点与解决方案
难点 根因分析 严重程度 解决方案
🕵️ JS反爬虫拦截 东方财富CDN对非浏览器请求返回JavaScript挑战页，Content-Type仍标注为PDF，导致首次运行17,400条全部失败 严重 调用 Node.js 执行挑战脚本，提取 Cookie 后携带重试；引入 asyncio.Lock 防止多协程竞争刷新 Cookie
📄 多格式文档 5,080条为 .docx，25条为HTML，4条为旧版.doc，pdfplumber全部报错且异常信息为空 严重 按 magic bytes 自动识别格式，分发到5个专用解析器（pdfplumber / python-docx / olefile / html.parser / UTF-8）
💰 API余额不足 DeepSeek账户在处理约7,000条时返回402错误，流水线中断 中等 充值后直接重新运行，断点续跑机制保证无数据损失
📦 超大PDF截断 部分PDF达20-29MB，30秒超时不足以完成下载 中等 改用 httpx 流式下载，读取超时从30秒提升至120秒
⚡ 空错误记录 Python 3.11+ 中 str(TimeoutError()) 返回空字符串，导致约3,661条 error_msg 为空，无法诊断 中等 改用 f"{type(e).__name__}: {e}"；重置为 pending 重跑，全部成功
🗜️ 不支持格式 1条RAR压缩包、1条47MB超大PDF（流式下载仍被截断） 轻微 标记为 skipped 状态，不计入失败，后续人工处理

评分Excel颜色含义典型公告类型
