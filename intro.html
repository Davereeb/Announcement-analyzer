A股公告情感分析系统

自动批量下载并解析 A 股上市公司公告，调用大语言模型完成四项核心分析任务

DeepSeek / Qwen LLM 20 并发异步处理 5 种文件格式 PDF · DOCX · DOC · HTML · 纯文本 断点续跑 SQLite + Excel 导出
系统概览
①
价值判断
判断公告是否含有影响股价的实质信息，输出 有 / 无
②
核心摘要
生成 50–100 字精炼摘要（无价值公告输出 null）
③
情感评分
量化评分 −2 到 +2（五级），反映公告对股价的情感方向
④
细粒度评分
估算全文中有效信息密度，输出 0–100 整数（百分比）

所有结果存入 SQLite 数据库，并导出带条件颜色格式的 Excel 报告。支持断点续跑：中途停止后重新运行，自动跳过已完成记录，不重复计费。

完整工作流程
1
读取输入 Excel

从 Excel 文件读取公告列表（股票代码、发布时间、公告标题、公告类型、公告链接），通过 INSERT OR IGNORE 批量写入 SQLite，已存在记录自动跳过，实现幂等导入。

2
查询待处理记录

从数据库取出状态为 pending 或可重试（failed 且 retry_count < 3）的记录，准备并发处理。

3
异步并发下载文档

使用 asyncio.Semaphore(20) 控制并发，通过 httpx 流式下载公告文件（120s 超时）。东方财富 PDF 链接需先通过 Node.js 执行 JS 反爬虫验证获取 Cookie。

4
自动识别格式并提取文本

通过文件 magic bytes 自动判断格式：%PDF → pdfplumber；PK → python-docx（.docx）；D0CF11E0 → olefile（.doc）；以 <html 开头 → html.parser；其余尝试 UTF-8 解码。

5
文本预处理与长文本采样

根据字数分档：short（<3,000字）全文直送；medium（3,000–10,000字）全文直送；long（>10,000字）采用三段采样策略（详见"长文本策略"章节）。

6
调用 LLM 分析

将处理后的文本连同公告标题、类型一起发送给 DeepSeek（或 Qwen 备用）。LLM 返回严格 JSON 格式，包含四个分析结果。失败自动重试，最多 3 次指数退避。

7
存库并导出 Excel

解析 JSON 响应，将所有字段写入 SQLite（状态更新为 done）。处理完成后运行 exporter.py，生成带五色色阶格式的 Excel 报告。

Prompt 设计要点
设计选择理由
强制纯 JSON 输出，禁止 ``` 标记方便正则兜底解析，避免 markdown 包裹导致 json.loads() 失败
在 prompt 内嵌入打分标准全文减少 LLM 自由发挥空间，确保评分边界一致，可重复
提供具体示例数值（granularity）帮助 LLM 校准密度感知，减少极端值偏差
is_valuable 先于 emotion 判断JSON 字段顺序引导 LLM 先做价值判断再打分，降低无价值公告被错误赋情感值的概率
temperature = 0.1在保留少量灵活性的同时，最大化评分稳定性
价值判断标准

LLM 判断公告是否含有影响股价的实质信息，输出 有 或 无。判断依据在 prompt 中明确列出，不依赖 LLM 自由裁量。

判断结果公告类型示例
有 业绩变动（预增/预减/报告期财务数据）、股东增减持、股份回购、重大中标/合同签署、重组并购、股权变更、分红转增、重大诉讼/仲裁、监管处罚、产品/资质获批、定增/配股方案
无 董事监事换届、会计师续聘、股东大会通知、章程修订、日常资金授权、问询函格式性回复（无新增实质内容）

注：价值判断决定后续字段——summary 和 emotion 在判为「无」时输出 null；granularity 无论有无价值均输出数值。

情感评分标准（−2 到 +2）

仅对判为「有价值」的公告打分。以下标准逐条写入 prompt，满足其一即可取对应分值。

+2
极度正面
  • 重大重组/并购完成且战略意义显著
  • 净利润/营收同比大幅增长（>100%）
  • 重大产品获批或核心资质取得
  • 大额股份回购完成且回购比例 >3%
+1
正面
  • 业绩同比增长但幅度适中（10%–100%）
  • 重大中标/签署合同且金额明确
  • 定增/再融资方案获批且募资顺利
  • 回购进展公告体现管理层信心
0
中性
  • 信息有实质内容但方向不明确
  • 资产出售/转让但对整体影响有限
  • 对外投资/设立子公司且规模较小
  • 调研活动纪要无明确业绩指引
−1
负面
  • 业绩下滑同比下降但未亏损
  • 募投项目进度延期且原因明确
  • 重要股东减持计划披露
  • 诉讼/仲裁立案且金额较小
−2
极度负面
  • 营收&净利润双降且由盈转亏
  • 净利润同比暴跌（>−80%）或巨额亏损
  • 重大违规/处罚涉及核心业务
  • 重大诉讼败诉且赔偿金额巨大

Excel 中情感评分列使用五色色阶：深绿（+2）→ 浅绿（+1）→ 白色（0）→ 浅红（−1）→ 深红（−2）

长文本处理策略

超过 10,000 字的文档（年报、招股书等）不全文送入 LLM，而是采用体感优化三段采样，确保覆盖关键披露区域，同时控制 token 成本。

████ 跳过前 8%（封面 / 目录 / 免责声明） ▓▓▓ 前段正文 body[0% → +2000字] ← 关键披露 / 摘要区 ···················· 省略 ···················· ▓▓▓ 中段正文 body[50% ± 1000字 → +2000字] ← 核心财务数据区 ···················· 省略 ···················· ▓▓▓ 后段正文 body[80% ± 1000字 → +2000字] ← 结论 / 风险前瞻区 ████ 跳过后 7%（附录 / 签字页 / 备查文件）
代码实现
def prepare_text(text: str, doc_type: str) -> str:
    if doc_type in ("short", "medium"):
        return text          # 全文直送
    n = len(text)
    body_start = int(n * 0.08)    # 跳过封面/目录
    body_end   = int(n * 0.93)    # 跳过附录/签字页
    body = text[body_start:body_end]
    blen = len(body)

    seg1 = body[:2000]                            # 前段
    seg2_s = blen // 2 - 1000
    seg2 = body[seg2_s : seg2_s + 2000]           # 中段
    seg3_s = int(blen * 0.80) - 1000
    seg3 = body[seg3_s : seg3_s + 2000]           # 后段

    return (
        "[节选·前段正文（已跳过封面目录）]\n" + seg1 + "\n\n"
        "[...省略...]\n\n"
        "[节选·中段正文]\n" + seg2 + "\n\n"
        "[...省略...]\n\n"
        "[节选·后段正文（已跳过附录签字页）]\n" + seg3
    )
系统模块
main.py

一键启动入口。顺序调用 pipeline → exporter，打印最终统计信息。

pipeline.py

异步流水线主逻辑。管理 asyncio.Semaphore 并发、进度条、Token 计数和费用估算。

fetcher.py

文档下载与文本提取。通过 magic bytes 自动识别格式，处理东方财富 JS 反爬，支持 PDF/DOCX/DOC/HTML/纯文本。

llm.py

LLM 调用封装。构建 System/User Prompt，调用 DeepSeek 或 Qwen API，解析 JSON 响应，指数退避重试。

db.py

SQLite 数据库操作层。维护状态机（pending → fetched → done/failed），支持幂等导入与断点续跑。

exporter.py

Excel 导出。为 emotion（5色）和 granularity（5色阶）字段应用条件颜色格式，自动时间戳命名。

记录状态机
pending  →  fetched  →  done
           ↘           ↘
            failed (retry_count < 3，下次自动重试)
            skipped（retry_count ≥ 3，永久跳过）
技术挑战与解决方案
🛡️
东方财富反爬虫（JS Challenge）

东方财富 PDF 下载链接需执行 JavaScript 计算才能获取有效 Cookie。解决方案：使用 Node.js 子进程执行 JS 代码，获取 Cookie 后复用；通过 asyncio.Lock 防止并发竞态。

📄
多格式文件自动识别

公告链接返回的文件格式不一（PDF/DOCX/DOC/HTML/纯文本），且 URL 中不一定有扩展名。解决方案：读取文件头部 magic bytes 进行二进制级格式判断，再选择对应解析库。

📏
长文本 Token 限制

年报、招股书等文档超过 10 万字，直接送入 LLM 会超出上下文限制且成本极高。解决方案：三段采样（前段+中段+后段各 2000 字），跳过封面/附录，覆盖关键信息区。

⚡
LLM 限流与并发控制

DeepSeek API 有 RPM/TPM 限制，过高并发触发 429 限流。解决方案：asyncio.Semaphore(20) 控制并发上限，失败后指数退避重试（最多 3 次），可在配置文件中调低并发数。

🔄
断点续跑与幂等性

大批量处理中途可能因网络或费用问题中断。解决方案：所有记录带状态字段，INSERT OR IGNORE 保证不重复导入，重启后只处理未完成记录，已成功记录不重复计费。

🔍
JSON 解析容错

LLM 偶尔在 JSON 前后附加多余文字。解决方案：先尝试 json.loads()，失败后用正则提取第一个 {...} 块再次解析，确保鲁棒性。
